# PhysX PTX Dual-Mode Build

## Overview

This document describes the **Elytar PTX dual-mode build system** for PhysX 5.6.1. It enables
compiling PhysX GPU code in two mutually exclusive modes, selectable at build time without
touching any PhysX source file:

| Mode | Description | Use case |
|------|-------------|----------|
| **Original mode** | Compile all 61 `.cu` files with `nvcc` as usual | Normal PhysX build |
| **PTX mode** | Replace every `.cu`'s device code with a pre-generated `.ptx` file | DSL integration, benchmarking |

In PTX mode the sanity-check PTX is generated by `nvcc -ptx` from the same `.cu` files, so the
build produces **functionally identical binaries with zero runtime overhead**. Once your
Triton-like DSL is ready, you swap each `<stem>.ptx` with your DSL-generated equivalent — the
rest of the build chain is unchanged.

## Quick Start

### Run in original mode (default)

```bash
./scripts/update_toolchain.sh
```

No extra steps needed. All 61 `.cu` files are compiled by `nvcc` exactly as NVIDIA intended.

### Run in PTX mode

```bash
# Step 1: Generate PTX from all 61 .cu files
PX_PTX_ARCH=compute_86 ./scripts/generate_ptx.sh --all

# Step 2: Build with PTX mode enabled
PX_USE_PTX_KERNELS=ON PX_PTX_ARCH=compute_86 ./scripts/update_toolchain.sh
```

For a single file during development:
```bash
./scripts/generate_ptx.sh --cu physx-5.6.1/source/gpusolver/src/CUDA/integration.cu
```

| GPU | Arch flag |
|-----|-----------|
| RTX 3090 | `compute_86` (default) |
| H200 | `compute_90` |

## Architecture: How PhysX Launches GPU Kernels

Understanding this is essential to see why the PTX mode works without overhead.

### PhysX does NOT use `<<<...>>>` syntax

All GPU kernels in PhysX 5.6.1 are launched through the **CUDA Driver API**, not the CUDA
Runtime API. The call chain is:

```
PxgSolver.cpp
  └─ mCudaContext->launchKernel("integrateCoreParallelLaunch", ...)
       └─ CudaContextManager::launchKernel()
            └─ cuModuleGetFunction(module, "integrateCoreParallelLaunch")  ← string lookup
                 └─ cuLaunchKernel(fn, ...)
```

Kernels are resolved **by name at runtime** through the `KernelWrangler`. This means the build
system only needs to make the correct compiled kernel code available under the right name —
it doesn't matter whether that code came from a `.cu` file or a `.ptx` file.

### Static registration of fat binaries

When `nvcc` compiles a `.cu` file it:
1. Embeds a **fat binary** (contains both PTX and SASS for the target architecture) inside the
   `.o` object file.
2. Generates static initializer stubs that call PhysX's custom interception hooks:
   - `PxGpuCudaRegisterFatBinary(void* fatBin)` — registers the module image, returns a handle
   - `PxGpuCudaRegisterFunction(int moduleIndex, const char* name)` — registers each kernel name

These hooks are implemented in `PxgPhysXGpu.cpp` and `CudaContextManager.cpp`. They populate
a module table (`s_moduleTable`) that `CudaContextManager::init()` later uses to call
`cuModuleLoadDataEx()` — loading each fat binary into a `CUmodule` on the device.

### The PTX mode replicates this exactly

In PTX mode, for each `.cu` file the auto-generated `<stem>_ptx_register.cpp` performs the
same two registrations at program startup via a C++ static initializer:

```cpp
// Inside anonymous namespace (internal linkage)
struct Elytar_integration_Registrar {
    Elytar_integration_Registrar() {
        void** handle = PxGpuCudaRegisterFatBinary(
            static_cast<void*>(integration_fatbin));   // our custom fatbin byte array
        int moduleIndex = static_cast<int>(
            reinterpret_cast<std::size_t>(handle));
        PxGpuCudaRegisterFunction(moduleIndex, "integrateCoreParallelLaunch");
        PxGpuCudaRegisterFunction(moduleIndex, "integrateKinematicParallelLaunch");
        // ... all kernels listed in the .ptx .entry directives
    }
};
static Elytar_integration_Registrar s_integration_registrar;
```

The fat binary is generated from the PTX by `nvcc -fatbin`, converted to a C byte array by
`bin2c`, and `#include`d as `integration_fatbin.h`. From the runtime's perspective this is
**indistinguishable** from the fat binary that `nvcc` would have embedded in the `.cu` object.

## Build Pipeline Comparison

### Original mode

```
integration.cu
    │
    ▼  nvcc (full .cu compilation: CUDA + host code together)
integration.o
    │  ┌─ embedded fatbin (PTX + SASS for target arch)
    │  ├─ __cudaRegisterFatBinary() stub (nvcc-generated static init)
    │  └─ __cudaRegisterFunction() stubs for each __global__ kernel
    │
    ▼  linker
libPhysXSolverGpu_static_64.a
    │
    ▼  runtime  cuModuleLoadDataEx(embedded_fatbin)
CUmodule  ──►  cuModuleGetFunction("integrateCoreParallelLaunch")  ──►  cuLaunchKernel
```

### PTX mode

```
integration.cu
    │
    ▼  nvcc -ptx  (scripts/generate_ptx.sh, run once before build)
integration.ptx                          ← stored in source tree: .../PTX/integration.ptx
    │
    ▼  nvcc -fatbin  (CMake add_custom_command, runs during cmake --build)
integration.fatbin
    │
    ▼  bin2c  (CMake add_custom_command, runs during cmake --build)
integration_fatbin.h                     ← C byte array: unsigned char integration_fatbin[]
    │
    ┌──────────────────────────────────────────────────────────────┐
    │  integration_ptx_register.cpp  (generated at cmake configure) │
    │  #include "integration_fatbin.h"                              │
    │  namespace { struct Elytar_integration_Registrar { ... }; }  │
    │  static Elytar_integration_Registrar s_integration_registrar; │
    └──────────────────────────────────────────────────────────────┘
    │
    ▼  g++ / clang++ (plain C++ compilation, no nvcc required)
integration_ptx_register.o
    │  ┌─ integration_fatbin data symbol (type D)
    │  └─ _GLOBAL__sub_I_integration_ptx_register.cpp  (static-init entry)
    │
    ▼  linker
libPhysXSolverGpu_static_64.a
    │
    ▼  runtime  (IDENTICAL path from here)
CUmodule  ──►  cuModuleGetFunction("integrateCoreParallelLaunch")  ──►  cuLaunchKernel
```

The two paths converge at `cuModuleLoadDataEx`. Everything from that point on — module
loading, function resolution, kernel dispatch — is **bit-for-bit identical**.

## Original vs PTX mode: Full Diff

| Aspect | Original mode | PTX mode |
|--------|--------------|----------|
| **What compiles the GPU code** | `nvcc` compiles `.cu` directly | `nvcc -ptx` then `nvcc -fatbin` |
| **When GPU compilation happens** | During `cmake --build` | PTX: before build. Fatbin: during build |
| **Who provides the fatbin** | `nvcc` embeds it in `.cu` object | `bin2c` converts `.fatbin` to C array |
| **Who calls `PxGpuCudaRegisterFatBinary`** | `nvcc`-generated static init stub | Auto-generated `_ptx_register.cpp` static init |
| **Who calls `PxGpuCudaRegisterFunction`** | `nvcc`-generated stub (one per kernel) | Auto-generated `_ptx_register.cpp` (names read from `.ptx`) |
| **Host init stub** (`initXxxKernelsN`) | Defined in `.cu` file itself | Auto-generated `_host_stub.cpp` (empty body) |
| **Linker input** | `.cu.o` objects from `nvcc` | `.cpp.o` objects from `g++/clang++` |
| **Symbol in `.a` for PTX fingerprint** | `__cuda_module_id` (per `.cu` TU) | `<stem>_fatbin` data symbol (type D) |
| **Runtime CUDA module loading** | `cuModuleLoadDataEx(fatbin)` | `cuModuleLoadDataEx(fatbin)` — identical |
| **Runtime kernel lookup** | `cuModuleGetFunction(name)` | `cuModuleGetFunction(name)` — identical |
| **Runtime kernel dispatch** | `cuLaunchKernel(fn, ...)` | `cuLaunchKernel(fn, ...)` — identical |
| **Kernel execution overhead** | baseline | **zero additional overhead** |
| **DSL swap** | Requires editing `.cu` source | Replace `.ptx` file, rebuild |

## Zero-Overhead Proof

**Claim:** PTX mode introduces zero kernel-execution overhead compared to original mode.

**Proof:**

1. **Same SASS at runtime.** `nvcc -fatbin -arch=sm_XX` compiles the PTX to SASS for the
   target architecture. The SASS in the original mode's fatbin and the PTX mode's fatbin is
   generated from the same source with the same flags — identical machine code.

2. **Same module loading.** Both modes call `cuModuleLoadDataEx()` with equivalent fat binary
   images. The CUDA Driver JIT-compiles PTX to SASS only if a matching SASS is absent, but
   since we use `nvcc -fatbin -arch=sm_XX` it always includes pre-compiled SASS — no JIT.

3. **Same kernel dispatch.** Both modes resolve kernels via `cuModuleGetFunction` by string
   name and call `cuLaunchKernel`. The function handle points to the same SASS instructions.

4. **No wrapper layer.** There are no extra function calls, indirections, or data copies in
   the hot path. The only difference is which `.o` file contributed the fatbin to the linker —
   this has zero effect at runtime.

**The only measurable difference** between original and PTX mode is a slightly longer
`cuModuleLoadDataEx` call at startup if the fatbin stores PTX-only for an architecture not
pre-compiled (JIT). Since we use `nvcc -fatbin -arch=sm_XX` this is always avoided.

## Implementation Details

### `ElytarPtxReplace.cmake` — the core macro

`physx-5.6.1/source/compiler/cmakegpu/ElytarPtxReplace.cmake` provides the
`ELYTAR_REPLACE_CU_WITH_PTX()` macro. For each `.cu` file it:

1. **Validates** that the corresponding `.ptx` file exists in the source tree (fails the cmake
   configure step with a clear error if missing).

2. **Parses the `.cu` file** for `extern "C" __host__ void <name>()` — the linker-force stub
   that `.cpp` files call to ensure the `.a` archive member is linked in. Generates
   `<stem>_host_stub.cpp` with an empty body of the same function.

3. **Reads kernel names from the `.ptx` file**, not from the `.cu` source. Every `.entry`
   directive in PTX is the exact string that `cuModuleGetFunction` / `PxGpuCudaRegisterFunction`
   must use. This handles both `extern "C"` kernels (plain names) and C++ kernels (mangled
   names like `_ZN5physx...`) transparently.

4. **Generates `<stem>_ptx_register.cpp`** using CMake's `file(GENERATE ...)` so the file is
   only written when content changes (avoids spurious rebuilds). The file contains:
   - `#include "<stem>_fatbin.h"` — the byte array from `bin2c`
   - A static initializer in an anonymous namespace that calls `PxGpuCudaRegisterFatBinary`
     then `PxGpuCudaRegisterFunction` for every kernel name extracted from the PTX
   - For modules with no `__global__` kernels (only `__device__` helpers), a `(void)moduleIndex;`
     line is inserted to silence the `-Wunused-variable` / `-Werror` build failure

5. **Registers two CMake custom commands**:
   - `nvcc -fatbin <stem>.ptx -arch=sm_XX -o <stem>.fatbin` — compiles PTX to a fat binary
   - `bin2c --name <stem>_fatbin <stem>.fatbin > <stem>_fatbin.h` — embeds it as C bytes

6. **Sets `OBJECT_DEPENDS`** on the generated `_ptx_register.cpp` to `<stem>_fatbin.h`, ensuring
   the C++ file is never compiled before the header it includes has been generated.

7. **Modifies the caller's CMake variable**: removes the `.cu` from the `KERNELS_VAR` list and
   appends the two generated `.cpp` files to `ELYTAR_PTX_EXTRA_SOURCES`.

### Kernel name extraction: why PTX, not `.cu`

An early version parsed `extern "C" __global__ void <name>(` from the `.cu` source. This
**failed** for most PhysX kernels because PhysX uses plain C++ linkage for many kernels — their
names are mangled by the compiler, and the `.cu` source does not have `extern "C"`. The PTX
`.entry` directive always records the exact symbol name, regardless of linkage — it is the
authoritative source.

### Anonymous namespace and symbol visibility

The `Elytar_<stem>_Registrar` struct lives inside an anonymous `namespace {}`, giving it
internal linkage. The C++ compiler does **not** export the struct name as a symbol; it inlines
the constructor into a file-scope static initializer (`_GLOBAL__sub_I_<stem>_ptx_register.cpp`).

This means `nm --defined-only | grep "Elytar_"` returns nothing even in a correctly-built PTX
archive. The correct PTX-mode fingerprints visible to `nm` are:
- **`<stem>_fatbin`** (type `D`) — the fatbin byte array from `bin2c`
- **`_GLOBAL__sub_I_<stem>_ptx_register.cpp`** (type `t`) — the static initializer

### `generate_ptx.sh` — hardened error handling

A critical bash subtlety: `set -e` is silently disabled inside functions invoked as the
condition of an `if` statement. Early versions of the script used `if process_cu_file ...; then`
patterns that masked `nvcc` failures, resulting in empty or invalid `.ptx` files being silently
written. The fix:

1. The function captures `nvcc`'s exit code explicitly:
   ```bash
   local nvcc_exit=0
   "${NVCC}" -ptx ... "${cu_file}" -o "${ptx_file}" || nvcc_exit=$?
   if [[ "${nvcc_exit}" -ne 0 ]]; then
       echo "ERROR: nvcc exited ${nvcc_exit}"; return 1
   fi
   ```
2. After a successful `nvcc` run, the script verifies that the generated `.ptx` contains at
   least one `.entry` directive — a zero-entry PTX would silently register zero kernels,
   causing `cuModuleGetFunction` to fail at runtime.

3. The flag `-Wno-unused-function` was removed from `CUDA_FLAGS`. This is a host compiler
   flag that must be passed via `-Xcompiler -Wno-unused-function` if needed; passing it bare
   causes `nvcc fatal: Unknown option` while allowing the script to continue (due to the
   `set -e` subtlety above), producing corrupt PTX files.

## Verification

After building in PTX mode, `update_toolchain.sh` calls `verify_ptx_mode()` which runs three
checks:

### Check 1 — PTX source files in the source tree

Counts `.ptx` files in each `source/<module>/src/PTX/` directory against the known totals:

| Module | Expected |
|--------|----------|
| `gpusolver` | 12 |
| `gpubroadphase` | 2 |
| `gpunarrowphase` | 25 |
| `gpusimulationcontroller` | 15 |
| `gpuarticulation` | 4 |
| `gpucommon` | 3 |
| **Total** | **61** |

### Check 2 — Auto-generated build stubs

Locates the `elytar_ptx/` directory under the build tree (searching up to 3 levels deep, since
the actual path is `<build>/sdk_gpu_source_bin/elytar_ptx/` not `<build>/elytar_ptx/`) and
verifies that 61 `*_ptx_register.cpp` and 61 `*.fatbin` files were generated.

### Check 3 — Symbol verification (`nm`)

Runs `nm --defined-only` on each GPU static library and checks for:
- **`<stem>_fatbin` data symbols** (type `D`) — present only in PTX-mode builds; each
  corresponds to one replaced `.cu` file's embedded byte array
- **`_GLOBAL__sub_I_*_ptx_register.cpp` constructor entries** — confirms the static
  initializers are present and will fire at process startup

Note: the struct name `Elytar_*_Registrar` does **not** appear in `nm` output because the
struct is in an anonymous namespace (internal linkage). The `*_fatbin` symbol is the correct
fingerprint to check.

## How to Add DSL-Generated PTX

When your Triton-like DSL is ready to replace a kernel:

1. Compile your DSL program to PTX targeting the appropriate architecture.
2. Ensure the `.entry` names in the PTX match the original kernel names. These names are
   what PhysX's kernel wrangler looks up via `cuModuleGetFunction`. You can find the expected
   names from the existing PTX files in the `PTX/` directories:
   ```bash
   grep '\.entry ' physx-5.6.1/source/gpusolver/src/PTX/integration.ptx
   ```
3. Place the PTX file at the expected path:
   ```
   physx-5.6.1/source/<module>/src/PTX/<stem>.ptx
   ```
4. Run `PX_USE_PTX_KERNELS=ON ./scripts/update_toolchain.sh`.

The CMake pipeline automatically picks up your file; no cmake changes are needed.

> **Tip:** The kernel name strings that PhysX uses for wrangler lookups are also visible in
> the kernel index headers, e.g., `PxgSolverKernelIndices.h`, `PxgNarrowphaseKernelIndices.h`.
> These must match the `.entry` names in your PTX exactly (case-sensitive).

## Troubleshooting

### "PTX file not found" during cmake configure

```
[Elytar] PTX file not found: .../integration.ptx
         Run: scripts/generate_ptx.sh --all
```

Run `./scripts/generate_ptx.sh --all` before cmake. All 61 `.ptx` files must exist in the
source tree before cmake can configure successfully in PTX mode.

### "bin2c not found" during cmake configure

`bin2c` ships with the CUDA toolkit alongside `nvcc`. Verify:
```bash
which bin2c || ls "$(dirname "$(which nvcc)")/bin2c"
```
If it's missing, your CUDA installation may be incomplete.

### PTX verification check 2 fails: "elytar_ptx/ not found"

The verification looks for `elytar_ptx/` up to 3 levels under the build directory using
`find`. If it still fails, cmake may not have run in PTX mode — check that `CMakeCache.txt`
contains `PX_USE_PTX_KERNELS:BOOL=ON`. Delete the cache and reconfigure if needed.

### PTX verification check 3 fails: "no *_fatbin symbols"

This means the PTX-mode stubs were not compiled into the archive. Possible causes:
- cmake ran in original mode (check the cache as above)
- The build system found stale `.o` files and didn't relink the archive. Run:
  ```bash
  cmake --build "${PHYSX_BUILD_DIR}" --clean-first
  ```

### Kernel name mismatch (DSL PTX)

If the wrangler cannot find your DSL kernel at runtime, check:
1. The kernel entry name in your `.ptx` — inspect with `grep '\.entry' your_kernel.ptx`
2. The name string in the kernel index header (e.g., `PxgSolverKernelIndices.h`)
3. The registration call in the auto-generated `_ptx_register.cpp` in the build tree —
   the macro reads names from the `.ptx` `.entry` directives, so if your DSL emits a
   different name than the original nvcc PTX, you need to match the original name.

### "unused variable 'moduleIndex'" compile error

This should not occur with the current implementation. If it does, it means
`_ptx_register.cpp` was generated by an older version of `ElytarPtxReplace.cmake`. Delete the
stale files and let cmake regenerate:
```bash
sudo rm -rf "${PHYSX_BUILD_DIR}/sdk_gpu_source_bin/elytar_ptx"
cmake --build "${PHYSX_BUILD_DIR}"
```

### Build order error (`<stem>_fatbin.h` not found at compile time)

The `OBJECT_DEPENDS` property on each `_ptx_register.cpp` source file ensures it is not
compiled until its `_fatbin.h` exists. If you see this error, the custom commands for
`nvcc -fatbin` and `bin2c` did not run first. A clean build resolves this:
```bash
cmake --build "${PHYSX_BUILD_DIR}" --clean-first
```
